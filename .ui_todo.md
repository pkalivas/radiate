  Brainstorm Directions

  Option A — Split AppState into role-scoped substates

  Break the god object into focused structs that get passed selectively:

  struct RunState { running, paused, index, last_render }
  struct DisplayState { tab, focus, modal, search }       // already exists, extend it
  struct MetricState { metrics, chart_state, tables }
  struct EvoState { front, objective, species, score }

  App owns all four; each panel only gets the substates it needs. The event router becomes the only place that reaches across all four. This is the lowest-risk refactor — it's largely additive and
   doesn't touch layout or rendering.

  ---
  Option B — Explicit modal/tab stack as a proper state machine

  Replace the "save prev_tab / prev_focus" pattern with a navigation stack:

  enum Screen {
      Dashboard(TabId),
      Modal(ModalKind, Box<Screen>),   // overlay + what's underneath
      Search(Box<Screen>),
  }

  open_modal(kind) pushes; close() pops. Escape always just pops. You can never accidentally lose the restore target. This is a mid-size refactor of DisplayState + the event handlers that
  currently call toggle_help, view_metric, etc.

  ---
  Option C — Generic TablePanel<Tag, Row, Key> to kill duplication

  The three table panels share everything except their tag filter, row builder, and AppTableState field. A single generic widget + a trait or closure could collapse all three:

  struct TablePanel<Row, Key> {
      title: &'static str,
      header: Vec<&'static str>,
      table_state_fn: fn(&mut AppState<C>) -> &mut AppTableState<Key>,
      row_builder: fn(&Metric) -> Option<Row>,
  }

  Or simpler: a MetricTable widget that takes tag + row-builder at construction time, with a PanelId pointing to the right AppTableState field. This is isolated enough that it doesn't require
  touching anything else.

  ---
  Option D — Decouple panels from the layout tree entirely

  Right now LayoutNode::draw() has a match on PanelId that manually instantiates each widget. Instead, register panels as a map:

  type PanelRenderer<C> = Box<dyn Fn(Rect, &mut Buffer, &mut AppState<C>)>;
  HashMap<PanelId, PanelRenderer<C>>

  The layout tree becomes purely structural (just knows PanelIds and constraints). New panels can be registered without touching template.rs. This is the highest-leverage refactor but also the
  most disruptive.

  ---
  Recommendation

  I'd start with A (split AppState) since it unblocks everything else — once each substate is scoped, it becomes much easier to do B or C without the god object getting in the way. Then tackle B
  for modal navigation correctness, since that's where real bugs can hide.

  C and D are nice-to-haves for maintainability but not urgent unless you're planning to add more panel types.

    AppState<C> went from one 560-line struct with 23 flat fields and 40+ methods to a four-field coordinator:

  pub struct AppState<C: Chromosome> {
      pub run: RunState,      // lifecycle: engine/ui/paused flags, render timing
      pub nav: NavState,      // navigation: tabs, focus panel, modal, search
      pub evo: EvoState<C>,   // evolution data: metrics, charts, species, front, pareto config
      pub tables: TableStates, // four table scroll states + selection logic
  }
